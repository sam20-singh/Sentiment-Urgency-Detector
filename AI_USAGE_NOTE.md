# AI Usage Note

This document satisfies the project submission requirement detailing how Artificial Intelligence was utilized during the development of **Sentinel — Sentiment & Urgency Detector**.

## 1. What AI Helped With

The AI assistant was instrumental in accelerating the development of both the frontend UI and the backend business logic:
- **FastAPI Boilerplate**: The AI quickly scaffolded the FastAPI application structure, including the Pydantic models, SQLite database connection logic (`database.py`), and the basic routing setup in `main.py`.
- **Vanilla Frontend & CSS**: The AI was responsible for writing the intricate CSS required for the "Obsidian & Electric Cyan" theme. It generated the math and logic for the CSS grid layout, the glassmorphism effects, the CSS keyframe animations (like the floating background blobs and sweeping button lasers), and the JavaScript `IntersectionObserver` that powers the scroll-reveal animations.
- **Groq Prompt Engineering**: The AI generated the strict JSON-mode prompt (`prompts.md`) used to instruct the Llama 3.3 70B model. This ensured the model would reliably output the required structured data (sentiment scores, urgency scores, tone, churn risk, and drafted replies) without hallucinating markdown wrappers.

## 2. What AI Got Wrong

While highly capable, the AI required manual course-correction in a few areas:
- **Z-Index Issues**: When initially adding a particle "snow" effect to the background, the AI set the z-index incorrectly, causing the glassmorphic background layer to completely obscure the particles. This required manual inspection of the stacking context in `style.css` and a subsequent fix to move the background blobs behind the particle canvas.
- **Pydantic Validation Quirks**: In early iterations, the AI attempted to define a Pydantic response model that did not perfectly align with the keys generated by the Groq JSON response, leading to `422 Unprocessable Entity` errors during testing. This was fixed by aligning the `AnalysisResponse` model strictly with the prompt schema.
- **Synchronous Execution Blockers**: The AI initially wrote the Discord Webhook alert logic synchronously within the `/analyze` route. This caused the API response to block while waiting for Discord to respond. It had to be refactored to use FastAPI's `BackgroundTasks` to execute the webhook asynchronously.

## 3. Best Prompts Used

The absolute most effective prompt utilized in this project was the core system prompt given to the Groq LLM to perform the ticket analysis. It utilizes strict behavioral constraints and schema enforcement:

```markdown
You are an expert customer support routing AI. Analyze the customer support ticket text and extract the exact information required.

1.  **sentiment_score**: Evaluate the sentiment from 1-10 (1 = extremely angry/frustrated, 10 = extremely happy/praising).
2.  **urgency_score**: Evaluate the urgency from 1-10 (1 = low priority/general question, 10 = critical system failure/blocker).
3.  **tone**: A 1-3 word description of the customer's tone (e.g., "Frustrated", "Neutral", "Anxious", "Polite").
4.  **is_churn_risk**: true if the customer is threatening to cancel, leave, or refund due to anger; otherwise false.
5.  **draft_reply**: A short, professional draft reply addressing their issue. Be empathetic if they are angry.

You MUST respond with ONLY valid, raw JSON. Do not include markdown formatting or backticks.

Expected Schema:
{
  "sentiment_score": integer,
  "urgency_score": integer,
  "tone": string,
  "is_churn_risk": boolean,
  "draft_reply": string
}
```
